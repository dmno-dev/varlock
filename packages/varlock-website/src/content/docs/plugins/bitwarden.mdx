Our Bitwarden plugin enables secure loading of secrets from **[Bitwarden Secrets Manager](https://bitwarden.com/products/secrets-manager/)** and from **[Bitwarden Password Manager](https://bitwarden.com/products/personal/)** / **[Vaultwarden](https://github.com/dani-garcia/vaultwarden)** (self-hosted) using declarative instructions within your `.env` files.

| | Bitwarden Secrets Manager | Bitwarden Password Manager / Vaultwarden |
|---|---|---|
| **Auth** | Machine account access token | CLI session token (`bw unlock`) |
| **Access** | REST API (no CLI needed) | `bw` CLI required |
| **Ideal for** | Production / CI environments | Local development |
| **Vaultwarden support** | ✗ | ✓ |

## Installation

In a JS/TS project, you may install the `@varlock/bitwarden-plugin` package as a normal dependency.
Otherwise you can just load it directly from your `.env.schema` file, as long as you add a version specifier.
See the [plugins guide](/guides/plugins/#installation) for more instructions on installing plugins.

## Bitwarden Secrets Manager

The Secrets Manager integration uses a machine account access token and communicates directly with the Bitwarden API — no CLI required. This makes it suitable for both **CI/CD and production environments**.

### Setup

```env-spec title=".env.schema"
# 1. Load the plugin
# @plugin(@varlock/bitwarden-plugin)
#
# 2. Initialize with your machine account access token
# @initBitwarden(accessToken=$BITWARDEN_ACCESS_TOKEN)
# ---

# 3. Add a config item for the access token
# @type=bitwardenAccessToken @sensitive
BITWARDEN_ACCESS_TOKEN=
```

### Machine account setup

<Steps>
1. **Create a machine account** in your Bitwarden organization

    Navigate to your Bitwarden organization's **Secrets Manager** → **Machine accounts** → Click **New machine account**.

    Provide a name (e.g., "Production App") and save it.

2. **Copy the access token** (displayed only once!)

    After creating the machine account, you'll see an **Access token**. Copy it immediately - it will only be displayed once.

    :::caution[Save the token securely]
    Store the access token securely. You won't be able to see it again after this step!
    :::

3. **Grant access to secrets**

    Grant your machine account access to the specific projects or secrets you need.

    **Via Projects (recommended):**
    - Create or select a project in Secrets Manager
    - Add secrets to the project
    - Grant your machine account access to the project

    **Direct secret access:**
    - Navigate to a specific secret → click **Access**
    - Add your machine account with "Can read" permissions

4. **Set your access token**

    Use your CI/CD system or platform's env var management to securely inject the `BITWARDEN_ACCESS_TOKEN` value.
</Steps>

:::tip[Permission levels]
Machine accounts can have _Can read_ (retrieve secrets only) or _Can read, write_ (retrieve, create, and edit secrets) permissions. For most use cases, _Can read_ is sufficient.
:::

### Self-hosted Bitwarden

For self-hosted Bitwarden instances, provide both the API and identity URLs:

```env-spec title=".env.schema"
# @initBitwarden(
#   accessToken=$BITWARDEN_ACCESS_TOKEN,
#   apiUrl="https://bitwarden.yourcompany.com/api",
#   identityUrl="https://bitwarden.yourcompany.com/identity"
# )
```

### Loading secrets

Fetch secrets by their UUID using the `bitwarden()` resolver:

```env-spec title=".env.schema"
# @plugin(@varlock/bitwarden-plugin)
# @initBitwarden(accessToken=$BITWARDEN_ACCESS_TOKEN)
# ---

# @type=bitwardenAccessToken @sensitive
BITWARDEN_ACCESS_TOKEN=

DATABASE_URL=bitwarden("12345678-1234-1234-1234-123456789abc")
API_KEY=bitwarden("87654321-4321-4321-4321-cba987654321")
```

:::tip[Finding secret UUIDs]
To find a secret's UUID, open Bitwarden Secrets Manager, navigate to the secret, and copy the UUID from the URL or secret details (format: `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`).
:::

### Multiple instances

Register multiple named instances to connect to different organizations or self-hosted servers:

```env-spec title=".env.schema"
# @initBitwarden(id=prod, accessToken=$PROD_ACCESS_TOKEN)
# @initBitwarden(id=dev, accessToken=$DEV_ACCESS_TOKEN)
# ---

PROD_SECRET=bitwarden(prod, "11111111-1111-1111-1111-111111111111")
DEV_SECRET=bitwarden(dev, "22222222-2222-2222-2222-222222222222")
```

## Bitwarden Password Manager / Vaultwarden

The Password Manager integration uses the official `bw` CLI tool. This makes it easy to use with **Vaultwarden** (the open-source self-hosted alternative) and regular Bitwarden password vault accounts.

:::note
Because this relies on the `bw` CLI and an interactive session token, it is primarily intended for **local development**. For production/CI use, prefer Bitwarden Secrets Manager (above) or another provider that supports non-interactive machine authentication.
:::

### Prerequisites

Install the [Bitwarden CLI](https://bitwarden.com/help/cli/):

<Tabs>
  <TabItem label="macOS">
    ```bash
    brew install bitwarden-cli
    ```
  </TabItem>
  <TabItem label="Linux">
    ```bash
    snap install bw
    ```
  </TabItem>
  <TabItem label="Windows">
    ```bash
    choco install bitwarden-cli
    ```
  </TabItem>
</Tabs>

#### Vaultwarden / self-hosted server

Point the CLI at your self-hosted server before logging in:

```bash
bw config server https://vaultwarden.yourcompany.com
```

### Setup + Auth

<Steps>
1. **Log in** (one-time setup):

   ```bash
   bw login
   ```

2. **Unlock** your vault and capture the session token:

   ```bash
   export BWP_SESSION=$(bw unlock --passwordenv BW_PASSWORD --raw)
   ```

   Or unlock interactively and copy the token it prints:

   ```bash
   bw unlock
   # export BW_SESSION="<token>"
   ```

3. **Configure** the plugin in your schema:

   ```env-spec title=".env.schema"
   # @plugin(@varlock/bitwarden-plugin)
   # @initBwp(sessionToken=$BWP_SESSION)
   # ---

   # @type=bwSessionToken @sensitive
   BWP_SESSION=
   ```
</Steps>

### Loading items

Use the `bwp()` resolver to fetch values from your vault. Items can be referenced by name or UUID.

```env-spec title=".env.schema"
# @plugin(@varlock/bitwarden-plugin)
# @initBwp(sessionToken=$BWP_SESSION)
# ---

# @type=bwSessionToken @sensitive
BWP_SESSION=

# Fetch the password field (default)
DATABASE_URL=bwp("My Database Item")

# Fetch specific built-in fields
DB_USER=bwp("My Database Item", field="username")
DB_NOTES=bwp("My Database Item", field="notes")
DB_TOTP=bwp("My Database Item", field="totp")
DB_URI=bwp("My Database Item", field="uri")

# Fetch a custom field
API_KEY=bwp("API Keys", field="production_api_key")

# Use UUID instead of item name
OTHER_SECRET=bwp("12345678-1234-1234-1234-123456789abc")
```

### Multiple instances

```env-spec title=".env.schema"
# @initBwp(id=work, sessionToken=$BWP_WORK_SESSION)
# @initBwp(id=personal, sessionToken=$BWP_PERSONAL_SESSION)
# ---

WORK_SECRET=bwp(work, "Work Item")
PERSONAL_SECRET=bwp(personal, "Personal Item", field="username")
```

## Reference

### Root decorators
<div class="reference-docs">
<div>
#### `@initBitwarden()`

Initialize a Bitwarden **Secrets Manager** plugin instance.

**Key/value args:**
- `accessToken` (required): Machine account access token. Should be a reference to a config item of type `bitwardenAccessToken`.
- `apiUrl` (optional): API URL for self-hosted Bitwarden (defaults to `https://api.bitwarden.com`)
- `identityUrl` (optional): Identity service URL for self-hosted Bitwarden (defaults to `https://identity.bitwarden.com`)
- `id` (optional): Instance identifier for multiple instances

```env-spec "@initBitwarden"
# @initBitwarden(accessToken=$BITWARDEN_ACCESS_TOKEN)
# ---
# @type=bitwardenAccessToken @sensitive
BITWARDEN_ACCESS_TOKEN=
```
</div>

<div>
#### `@initBwp()`

Initialize a Bitwarden **Password Manager / Vaultwarden** plugin instance (requires the `bw` CLI).

**Key/value args:**
- `sessionToken` (required): CLI session token from `bw unlock`. Should be a reference to a config item of type `bwSessionToken`.
- `id` (optional): Instance identifier for multiple instances

```env-spec "@initBwp"
# @initBwp(sessionToken=$BWP_SESSION)
# ---
# @type=bwSessionToken @sensitive
BWP_SESSION=
```
</div>
</div>

### Data types
<div class="reference-docs">
<div>
#### `bitwardenAccessToken`

Represents a Bitwarden Secrets Manager machine account access token. Validation ensures the token is in the correct format (`0.<client_id>.<client_secret>:<encryption_key>`).
The type itself is marked as `@sensitive`, so adding an explicit `@sensitive` decorator is optional.

```env-spec "bitwardenAccessToken"
# @type=bitwardenAccessToken
BITWARDEN_ACCESS_TOKEN=
```
</div>

<div>
#### `bitwardenSecretId`

Represents a secret UUID in Bitwarden Secrets Manager. Validation ensures the ID is a valid UUID format.

```env-spec "bitwardenSecretId"
# @type=bitwardenSecretId
MY_SECRET_ID=12345678-1234-1234-1234-123456789abc
```
</div>

<div>
#### `bitwardenOrganizationId`

Represents an organization UUID in Bitwarden. Validation ensures the ID is a valid UUID format.

```env-spec "bitwardenOrganizationId"
# @type=bitwardenOrganizationId
BITWARDEN_ORG_ID=87654321-4321-4321-4321-cba987654321
```
</div>

<div>
#### `bwSessionToken`

Represents a Bitwarden CLI session token obtained from `bw unlock`. The type itself is marked as `@sensitive`.

```env-spec "bwSessionToken"
# @type=bwSessionToken
BWP_SESSION=
```
</div>
</div>

### Resolver functions
<div class="reference-docs">
<div>
#### `bitwarden()`

Fetch a secret from Bitwarden **Secrets Manager** by UUID.

**Array args:**
- `instanceId` (optional): instance identifier to use when multiple plugin instances are initialized
- `secretId` (required): secret UUID in the format `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`

```env-spec /bitwarden\\(.*\\)/
# Fetch by secret UUID
DATABASE_URL=bitwarden("12345678-1234-1234-1234-123456789abc")

# With instance ID
PROD_SECRET=bitwarden(prod, "11111111-1111-1111-1111-111111111111")
```
</div>

<div>
#### `bwp()`

Fetch a field value from a Bitwarden **Password Manager / Vaultwarden** vault item via the `bw` CLI.

**Array args:**
- `instanceId` (optional): instance identifier for multiple plugin instances
- `itemName` (required): item name or UUID

**Key/value args:**
- `field` (optional): field to read (defaults to `password`). Built-in fields: `password`, `username`, `notes`, `totp`, `uri`. Any other value is looked up as a custom field (case-insensitive).

```env-spec /bwp\\(.*\\)/
# Default (password field)
DATABASE_URL=bwp("My DB Item")

# Specific field
DB_USER=bwp("My DB Item", field="username")

# Custom field
API_KEY=bwp("API Keys", field="production_api_key")

# Named instance
WORK_SECRET=bwp(work, "Work Item", field="notes")
```
</div>
</div>

## Troubleshooting

### Secrets Manager

**Secret not found**
- Verify the secret UUID is correct (must be valid UUID format)
- Check that the secret exists in your Bitwarden Secrets Manager
- Ensure your machine account has access to the secret or its project

**Authentication failed**
- Verify the access token is correct and has not been revoked or expired
- Ensure the machine account is not disabled
- For self-hosted: verify `apiUrl` and `identityUrl` are correct

**Invalid UUID format**
- Secret IDs must be valid UUIDs: `12345678-1234-1234-1234-123456789abc`

### Password Manager / Vaultwarden

**`bw` CLI not found**
- Install the Bitwarden CLI: https://bitwarden.com/help/cli/
- Ensure `bw` is available in your `$PATH`

**Session invalid or expired**
- Run `bw unlock` again to get a fresh session token
- Update your `BWP_SESSION` (or whichever env var you configured) with the new token

**Item not found**
- Verify the item name matches exactly (or use its UUID)
- Run `bw list items` to see all items in your vault
- Make sure your vault is synced: `bw sync`
