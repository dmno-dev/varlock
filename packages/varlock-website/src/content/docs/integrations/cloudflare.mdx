Varlock provides a robust solution for managing environment variables in Cloudflare Workers, offering validation, type safety, and security features that go beyond Cloudflare's built-in environment variable handling.

Both approaches below use `varlock-wrangler` — a thin wrapper around `wrangler` that automatically resolves and uploads your env vars as Cloudflare secrets and vars during deployment, and injects them into miniflare during local dev.

## Two approaches

1. **Using `varlock-wrangler`** - A drop-in replacement for `wrangler` with a one-line init import in your worker
2. **With the Vite plugin** - If you're already using the [Cloudflare Workers Vite plugin](https://developers.cloudflare.com/workers/vite-plugin/), you can add the varlock Cloudflare Vite plugin for automatic build-time integration

## Approach 1: Using `varlock-wrangler`

Use `varlock-wrangler` as a drop-in replacement for `wrangler`, and initialize varlock in your worker code with a single import.

### Setup

<Steps>

1. **Install packages**
    <InstallJsDepsWidget packages="@varlock/cloudflare-integration varlock" />

1. **Run `varlock init` to set up your `.env.schema` file**

    This will guide you through setting up your `.env.schema` file, based on your existing `.env` file(s). Make sure to review it carefully.

    <ExecCommandWidget command="varlock init" showBinary={false} />

1. **Add the varlock init import to your worker entry point**

    This reads the `__VARLOCK_ENV` binding, initializes the varlock `ENV` proxy, and applies console redaction and response leak detection:

    ```diff lang="ts" title="src/index.ts"
    +import '@varlock/cloudflare-integration/init';
    +import { ENV } from 'varlock/env';

    export default {
      async fetch(request, env, ctx) {
    +    // both work:
    +    console.log(ENV.MY_VAR);  // varlock
    +    console.log(env.MY_VAR);  // native Cloudflare
        return new Response('Hello!');
      },
    };
    ```

1. **Update your package.json scripts to use `varlock-wrangler`**

    ```diff lang="json" title="package.json" add="varlock-"
    {
      "scripts": {
    -    "dev": "wrangler dev",
    +    "dev": "varlock-wrangler dev",
    -    "deploy": "wrangler deploy",
    +    "deploy": "varlock-wrangler deploy"
    -    "types": "wrangler types",
    +    "types": "varlock-wrangler types",
      }
    }
    ```

</Steps>

### How it works

- **In dev**: `varlock-wrangler dev` resolves your env and injects it into miniflare via a [Unix named pipe](https://en.wikipedia.org/wiki/Named_pipe) — secrets are never written to disk as plaintext files. It also watches your `.env` files and automatically restarts wrangler when they change.
- **In production**: `varlock-wrangler deploy` uploads non-sensitive values as Cloudflare vars and sensitive values as Cloudflare secrets. The worker reads them at runtime via `import { env } from 'cloudflare:workers'`.

## Approach 2: With the Vite plugin

If you're already using the [Cloudflare Workers Vite plugin](https://developers.cloudflare.com/workers/vite-plugin/), you can add the varlock Cloudflare Vite plugin for automatic build-time integration — no init import needed.

### Setup

<Steps>

1. **Install packages**
    <InstallJsDepsWidget packages="@varlock/cloudflare-integration varlock" />

1. **Run `varlock init` to set up your `.env.schema` file**

    <ExecCommandWidget command="varlock init" showBinary={false} />

1. **Update your Vite config**

    Add `varlockCloudflareVitePlugin` to your Vite config:

    ```diff lang="ts" title="vite.config.ts"
    import { defineConfig } from 'vite';
    import { cloudflare } from '@cloudflare/vite-plugin';
    +import { varlockCloudflareVitePlugin } from '@varlock/cloudflare-integration';

    export default defineConfig({
      plugins: [
    +    varlockCloudflareVitePlugin(),
        cloudflare(),
        // other plugins ...
      ],
    });
    ```

1. **Update your deploy script to use `varlock-wrangler`**

    ```diff lang="json" title="package.json"
    {
      "scripts": {
        "dev": "vite dev",
        "build": "vite build",
    -    "deploy": "npm run build && wrangler deploy",
    +    "deploy": "npm run build && varlock-wrangler deploy"
      }
    }
    ```

</Steps>

During `vite dev`, the plugin automatically loads and validates your env. When you deploy, `varlock-wrangler` uploads sensitive values as Cloudflare secrets and non-sensitive values as Cloudflare vars.

### How it works

- **In dev**: The Vite plugin bundles resolved env directly into the worker code (via `resolved-env` mode), so everything works locally without Cloudflare secret bindings.
- **In production**: The worker reads the `__VARLOCK_ENV` secret from Cloudflare's bindings at runtime via `import { env } from 'cloudflare:workers'`. Individual vars are also available natively via `env.MY_VAR`.

### Enabling native `env` access in dev

By default with Vite, native Cloudflare `env.MY_VAR` access only works in production (where the values are uploaded as secrets/vars). To also make it work in local dev, you can inject resolved vars into miniflare's bindings via the cloudflare plugin's `config` option:

```diff lang="ts" title="vite.config.ts"
import { defineConfig } from 'vite';
import { cloudflare } from '@cloudflare/vite-plugin';
-import { varlockCloudflareVitePlugin } from '@varlock/cloudflare-integration';
+import { varlockCloudflareVitePlugin, resolvedEnvVars } from '@varlock/cloudflare-integration';

export default defineConfig({
  plugins: [
    varlockCloudflareVitePlugin(),
-    cloudflare(),
+    cloudflare({
+      config: (cfg) => ({ vars: { ...cfg.vars, ...resolvedEnvVars() } }),
+    }),
  ],
});
```

This is optional — `ENV.MY_VAR` via varlock always works without this.

### Upgrading from `@varlock/vite-integration`

If you were previously using `varlockVitePlugin()` from `@varlock/vite-integration` with Cloudflare Workers, your secrets were being bundled directly into the worker code. To upgrade:

1. Install `@varlock/cloudflare-integration` (you can remove `@varlock/vite-integration` — it's included as a dependency)
2. In `vite.config.ts`, replace `varlockVitePlugin()` with `varlockCloudflareVitePlugin()`
3. Replace `wrangler deploy` with `varlock-wrangler deploy` in your deploy script

```diff lang="ts" title="vite.config.ts"
-import { varlockVitePlugin } from '@varlock/vite-integration';
+import { varlockCloudflareVitePlugin } from '@varlock/cloudflare-integration';

export default defineConfig({
  plugins: [
-    varlockVitePlugin(),
+    varlockCloudflareVitePlugin(),
    cloudflare(),
  ],
});
```

Your secrets will no longer be bundled into the JS artifact — they'll be stored in Cloudflare's secret management instead.

## Accessing environment variables

You can access your environment variables using varlock's `ENV` object, or using Cloudflare's native `env` bindings:

```ts title="src/index.ts"
// ✅ Recommended - uses varlock's ENV (always works)
import { ENV } from 'varlock/env';
console.log(ENV.API_KEY);

// ✅ Also works - native Cloudflare access
import { env } from "cloudflare:workers";
console.log(env.API_KEY);
```

## `varlock-wrangler` CLI

`varlock-wrangler` is a drop-in replacement for `wrangler`. It enhances the following commands:

### `varlock-wrangler dev`

Wraps `wrangler dev` with automatic env injection:

1. Resolves your environment variables
2. Injects them into miniflare via a **Unix named pipe** (FIFO) — secrets are never written to disk as plaintext
3. Watches your `.env` files and restarts wrangler automatically when they change

### `varlock-wrangler deploy` / `versions upload`

1. Resolves your environment variables using your `.env.schema` and `.env` files
2. Uploads **non-sensitive** values as Cloudflare vars (`--var` flags)
3. Uploads **sensitive** values as Cloudflare secrets (`--secrets-file`)
4. Includes a `__VARLOCK_ENV` secret containing the full resolved env graph for the varlock runtime

The secrets file is written to a temp directory, used for the deploy, and immediately deleted — secrets are never left on disk.

### `varlock-wrangler types`

Generates Cloudflare Worker types (the `Env` interface) including all varlock-managed environment variables. This ensures your `env.MY_VAR` access is properly typed alongside other Cloudflare bindings (KV, D1, etc.).

```bash
varlock-wrangler types
```

### Other commands

All other wrangler commands (e.g., `tail`, `secret list`) are passed through to `wrangler` unchanged.

## Managing Multiple Environments

Varlock can load multiple _environment-specific_ `.env` files (e.g., `.env.development`, `.env.preview`, `.env.production`) by using the [`@currentEnv` root decorator](/reference/root-decorators/#currentenv).

If you are using Cloudflare's CI, you can use the current branch name (`WORKERS_CI_BRANCH`) to determine the environment:

```env-spec title=".env.schema"
# @currentEnv=$APP_ENV
# ---
WORKERS_CI_BRANCH=
# @type=enum(development, preview, production, test)
APP_ENV=remap($WORKERS_CI_BRANCH, "main", production, regex(.*), preview, undefined, development)
```

For more information, see the [environments guide](/guides/environments).
