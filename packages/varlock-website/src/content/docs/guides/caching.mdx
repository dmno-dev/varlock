Varlock has a built-in caching system, secured by [local encryption](/guides/local-encryption/).

Rather than caching environment variables values, it is general key-value store, usable by plugins to cache arbitrary internal requests, or by an explicit [`cache()`](/reference/functions/#cache) function to cache any resolver.

## Plugin cache behavior

Plugins have access to cache helpers that can be used to avoid repeated network round trips, or to hold short lived tokens. They will usually expose params (like `cacheTtl`) to let the user opt-into caching. This will still respect global cache mode and cli flags. Each plugin can decide how to cache things and it will often depend on the shape of the external API.

## Common `cache()` use case: stable local-dev random values

The most common explicit use of `cache()` is generating random values once for local development, then reusing them so they stay stable across restarts.

```env-spec title=".env.local"
# Good for local-only generated values
SESSION_SECRET=cache(randomHex(32))
INSTANCE_ID=cache(randomUuid())
```

This allows a new development environment to generate values, while keeping them stable.

## Global cache disable

Caching is mostly useful for local development, when there may be many invocations of varlock, and we have biometric encryption available. So default behavior is to disable caching in CI and when local encryption is using the basic file-based fallback.

However if you need to explcitly enable/disable caching, you can use the [`@cache`](/reference/root-decorators/#cache) root decorator.

## Per-invocation CLI controls

Both [`varlock load`](/reference/cli-commands/#load) and [`varlock run`](/reference/cli-commands/#run) support:

- `--skip-cache`: disables cache reads and writes for that invocation (overrides `@cache`)
- `--clear-cache`: clears the active cache store before resolving values

If both are used, cache is cleared first, then skipped for the rest of the run.

## Inspecting and clearing cache

Use [`varlock cache`](/reference/cli-commands/#cache) to inspect and clear the **disk cache**:

```bash
varlock cache status
varlock cache clear --yes
varlock cache clear --plugin 1password --yes
```

Notes:

- This command manages disk cache entries only.
- In `memory` mode, entries are process-local and are not visible via `varlock cache`.

## Troubleshooting

### "I changed a resolver but still see old values"

- If you used a custom `key`, that key may intentionally pin the cache entry.
- Run with `--clear-cache` or clear entries via `varlock cache clear`.

### "`varlock cache` shows nothing, but caching seems active"

- You are likely in `memory` mode (for example in CI or file-backend fallback mode).

### "I want no cache artifacts on disk"

- Use `@cache=memory` (or `@cache=disabled`) and avoid `disk` mode.
