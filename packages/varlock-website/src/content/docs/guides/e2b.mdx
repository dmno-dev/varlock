---

[E2B](https://e2b.dev) runs code in cloud microVM sandboxes, commonly as the execution layer for AI agents. Sandboxes get their env vars from the code that creates them (`envs` at sandbox creation or per command), which makes that orchestrator code the natural place for varlock to do its job.

## Choosing an approach

| Approach | When to use | Where real secrets live |
| --- | --- | --- |
| **[Pass resolved values](#passing-resolved-values)** | Trusted workloads; you want validation and resolution without changing how sandboxes consume env | Orchestrator and every sandbox |
| **[varlock inside the sandbox](#running-varlock-inside-the-sandbox)** | The sandboxed project itself uses varlock, or should resolve its own secrets via a [plugin](/guides/plugins/) | Inside the sandbox |
| **[Credential proxy](#credential-proxy-keeping-secrets-out-of-sandboxes)** | Agentic or untrusted workloads that should never hold real credentials | Broker only (a dedicated sandbox, or your machine) |

The approaches compose: most setups start by passing resolved values, and move agent workloads to the proxy.

## Passing resolved values

Run the orchestrator under `varlock run` and pass values through. Varlock resolves them (from plugins, `.env.local`, etc.), validates them against your schema, and redacts them in the orchestrator's logs; E2B delivery stays plain `envs`:

```bash
varlock run -- node orchestrate.ts
```

```ts title="orchestrate.ts"
import { Sandbox } from 'e2b';

const sandbox = await Sandbox.create({
  envs: {
    STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY!,
  },
});
```

This is the standard E2B posture: sandboxes hold real values, and the values transit E2B's API. That is fine for workloads you trust with the credentials they use. For agents, see the [proxy section](#credential-proxy-keeping-secrets-out-of-sandboxes).

## Running varlock inside the sandbox

If the code running in the sandbox is itself a varlock project (it has a `.env.schema`), install varlock in the sandbox and use `varlock run` there, the same way you would in a [Docker container](/guides/docker/):

```ts
await sandbox.commands.run('curl -sSfL https://varlock.dev/install.sh | sh -s');
await sandbox.commands.run(
  'cd /home/user/proj && /home/user/.config/varlock/bin/varlock run -- node app.js',
  { envs: { STRIPE_SECRET_KEY: process.env.STRIPE_SECRET_KEY! } },
);
```

Values can come in via `envs` as above (the schema validates them at load), or the sandbox can resolve its own secrets with a [plugin](/guides/plugins/) and a service-account token passed in as the only bootstrap credential. Bake varlock into a [custom template](https://e2b.dev/docs/template/quickstart) to skip the install step at runtime.

## Credential proxy: keeping secrets out of sandboxes

Both approaches above put real values inside sandboxes. For agentic workloads, the [varlock credential proxy](/guides/proxy/) removes them entirely: sandboxes hold only [placeholders](/guides/proxy/#placeholders), and the proxy injects real values into requests at the wire, on verified TLS connections to hosts your schema allows. A compromised or prompt-injected agent can exfiltrate nothing but placeholders, and every request is checked against your [`@proxy` rules](/guides/proxy/#routing-rules) and recorded in the [audit log](/guides/proxy/#auditing).

E2B sandboxes are remote, so they cannot reach a proxy listening on your machine's loopback. The bridge is a small WebSocket tunnel ([chisel](https://github.com/jpillora/chisel)), which E2B's public sandbox URLs carry fine. There are two shapes, depending on where you want the proxy (and your real secrets) to live:

- **[Broker sandbox](#broker-sandbox)**: the proxy runs in one dedicated E2B sandbox; agent sandboxes tunnel to it. Fits orchestrators and fleets, where there is no long-lived machine of your own in the loop.
- **[Proxy on your machine](#proxy-on-your-machine)**: the proxy stays local and secrets never leave your machine; a minimal relay sandbox is the rendezvous point. Fits the dev loop.

Both shapes leave the proxy listening on loopback at every hop, which is exactly how it wants to be deployed. The tunnel is what makes the loopback ends meet.

### Schema setup

Mark the secrets your agents use with [`@proxy(domain=...)`](/reference/item-decorators/#proxy) and give each one an explicit [`@placeholder`](/reference/item-decorators/#placeholder):

```env-spec title=".env.schema"
# @proxyConfig={egress="strict"}
# ---

# @proxy(domain="api.anthropic.com")
# @placeholder=sk-ant-api03-000000000000000000000000
ANTHROPIC_API_KEY=

# @proxy(domain="api.stripe.com")
# @placeholder=sk_test_00000000000000000000000000
STRIPE_SECRET_KEY=
```

Explicit placeholders matter here: your orchestrator code passes them to agent sandboxes via `envs`, so they need to be values you know up front rather than generated ones.

[`@proxyConfig={egress="strict"}`](/guides/proxy/#egress-modes) blocks any proxied request that does not match a rule. Start with the default permissive mode if your agents need broad internet access through the proxy, and tighten later.

You will also need an [E2B API key](https://e2b.dev/docs) for the orchestrator that creates sandboxes. It can be a varlock-managed secret like any other.

### Broker sandbox

One long-lived sandbox (the broker) runs `varlock proxy start` plus a chisel server. Agent sandboxes run a chisel client that gives them a local port which lands on the broker's loopback proxy.

```
[agent sandbox]                     [broker sandbox]
  your agent                          varlock proxy :8080 (loopback)
  HTTPS_PROXY=127.0.0.1:8888   ws    chisel server :8000
  chisel client  ─────────────────▶  (public getHost URL)
```

#### Start the broker

```ts title="orchestrate.ts"
import { Sandbox } from 'e2b';

const CHISEL_URL = 'https://github.com/jpillora/chisel/releases/download/v1.10.1/chisel_1.10.1_linux_amd64.gz';
const TUNNEL_AUTH = `varlock:${crypto.randomUUID()}`;
const VARLOCK = '/home/user/.config/varlock/bin/varlock';

const broker = await Sandbox.create({ timeoutMs: 60 * 60 * 1000 });

// install varlock + chisel
await broker.commands.run('curl -sSfL https://varlock.dev/install.sh | sh -s');
await broker.commands.run(`curl -sSL ${CHISEL_URL} | gunzip > /tmp/chisel && chmod +x /tmp/chisel`);

// schema + real values
await broker.files.write('/home/user/proj/.env.schema', envSchemaContents);
await broker.files.write('/home/user/proj/.env.local', `ANTHROPIC_API_KEY=${realAnthropicKey}\nSTRIPE_SECRET_KEY=${realStripeKey}\n`);

// start the proxy (loopback) and wait for its CA certs
await broker.commands.run(
  `cd /home/user/proj && ${VARLOCK} proxy start --port 8080 --cert-dir /home/user/proxy-certs > /home/user/proxy.log 2>&1`,
  { background: true, timeoutMs: 0 },
);
await broker.commands.run('until test -f /home/user/proxy-certs/combined-ca.pem; do sleep 0.2; done', { timeoutMs: 30_000 });

// expose the tunnel server on the broker's public URL
await broker.commands.run(
  `/tmp/chisel server --host 0.0.0.0 --port 8000 --auth '${TUNNEL_AUTH}' > /home/user/chisel.log 2>&1`,
  { background: true, timeoutMs: 0 },
);
const brokerHost = broker.getHost(8000);

// pull the CA certs out so agents can trust the proxy
const caCert = await broker.files.read('/home/user/proxy-certs/ca-cert.pem');
const combinedCa = await broker.files.read('/home/user/proxy-certs/combined-ca.pem');
```

Writing `.env.local` from the orchestrator is the simplest bootstrap. If you would rather resolve secrets inside the broker, install a [plugin](/guides/plugins/) and provide its service-account token instead; either way there is one bootstrap credential for the broker, instead of real secrets in every agent sandbox.

#### Start agent sandboxes

Each agent gets the standard proxy wiring (the same env vars `varlock proxy run` injects locally), the CA certs, the placeholders, and a chisel client:

```ts title="orchestrate.ts (continued)"
const agent = await Sandbox.create({
  timeoutMs: 60 * 60 * 1000,
  envs: {
    HTTP_PROXY: 'http://127.0.0.1:8888',
    HTTPS_PROXY: 'http://127.0.0.1:8888',
    NO_PROXY: 'localhost,127.0.0.1,::1',
    NODE_EXTRA_CA_CERTS: '/home/user/certs/ca-cert.pem',
    SSL_CERT_FILE: '/home/user/certs/combined-ca.pem',
    REQUESTS_CA_BUNDLE: '/home/user/certs/combined-ca.pem',
    CURL_CA_BUNDLE: '/home/user/certs/combined-ca.pem',
    GIT_SSL_CAINFO: '/home/user/certs/combined-ca.pem',
    // placeholders only - real values exist only in the broker
    ANTHROPIC_API_KEY: 'sk-ant-api03-000000000000000000000000',
    STRIPE_SECRET_KEY: 'sk_test_00000000000000000000000000',
  },
});

await agent.commands.run(`curl -sSL ${CHISEL_URL} | gunzip > /tmp/chisel && chmod +x /tmp/chisel`);
await agent.files.write('/home/user/certs/ca-cert.pem', caCert);
await agent.files.write('/home/user/certs/combined-ca.pem', combinedCa);
await agent.commands.run(
  `/tmp/chisel client --auth '${TUNNEL_AUTH}' https://${brokerHost} 8888:127.0.0.1:8080 > /home/user/chisel.log 2>&1`,
  { background: true, timeoutMs: 0 },
);

// now run your agent workload; its egress goes through the broker's policy
await agent.commands.run('your-agent-command');
```

Baking varlock and chisel into a [custom template](https://e2b.dev/docs/template/quickstart) instead of installing at startup makes sandbox creation faster; the wiring is the same.

To see what agents are doing, run [`varlock proxy audit`](/reference/cli-commands/#proxy) (or `proxy status --watch`) inside the broker: every request records its host, path, decision, and which keys were injected.

#### Lock down agent egress

So far the proxy governs proxied traffic, but an agent could still make direct connections that bypass `HTTPS_PROXY`. Those requests carry placeholders at worst, so no secrets are at stake, but E2B can close the gap entirely with its [network egress rules](https://e2b.dev/docs/sandbox/internet-access): restrict the agent to the broker's URL and every request has to go through varlock policy.

Apply the restriction **after** provisioning, not at `Sandbox.create`. The agent needs open egress briefly to download chisel (skip that step by baking chisel into your template), and the clamp does not disturb the established tunnel:

```ts
await agent.updateNetwork({
  allowOut: [brokerHost],
  denyOut: ['0.0.0.0/0'],
});
```

This is enforced by E2B's infrastructure outside the VM, so nothing the agent does from inside can lift it. It is the same role the internal-only network plays in the [`--sandbox=docker`](/guides/sandboxing/#container---sandboxdocker) recipe, with a stronger enforcement point.

#### Trust model

Be clear-eyed about what this shape protects against. The broker holds real secrets inside E2B's cloud, so E2B's infrastructure is inside your trust boundary, same as it would be for secrets passed to any sandbox. What changes is the blast radius on your side: agents never hold secrets, so a compromised agent sandbox yields placeholders, an authenticated tunnel endpoint, and only whatever requests your rules and egress mode allow. Rotation, policy, and audit live in one place instead of N sandboxes.

Two practical notes:

- No human is attached to the broker, so treat its policy as static: allow rules, `block` rules, `@proxy=omit`, and strict egress. To change policy, [edit the schema and reload](/guides/proxy/#editing-the-schema-while-a-session-is-running) from the orchestrator, or restart the broker.
- Agents hold the tunnel credential, deliberately. Tunnel auth exists to keep the rest of the internet off your broker's public URL, not to keep agents out; the proxy's rules are the gate that applies to agents.

### Proxy on your machine

For local development, keep the proxy (and your secrets, resolver plugins, and biometric unlock) on your machine. Your laptop is not reachable from E2B's cloud, so a minimal relay sandbox stands in the middle: chisel in reverse mode lets your machine dial out to the relay and publish its loopback proxy there, and agents connect to the relay exactly as they connect to a broker.

```
[agent sandbox]            [relay sandbox]              [your machine]
  your agent                 chisel server :8000          varlock proxy :8080
  chisel client ── ws ──▶    --reverse       ◀── ws ──   chisel client (R:8080)
```

The relay never sees plaintext secrets: TLS runs end to end between the agent's client and the varlock proxy on your machine, so the relay carries ciphertext.

Start the relay (`--reverse` is the only difference from the broker's server):

```ts
const relay = await Sandbox.create({ timeoutMs: 60 * 60 * 1000 });
await relay.commands.run(`curl -sSL ${CHISEL_URL} | gunzip > /tmp/chisel && chmod +x /tmp/chisel`);
await relay.commands.run(
  `/tmp/chisel server --reverse --host 0.0.0.0 --port 8000 --auth '${TUNNEL_AUTH}' > /home/user/chisel.log 2>&1`,
  { background: true, timeoutMs: 0 },
);
const relayHost = relay.getHost(8000);
```

On your machine, start the proxy and reverse-publish it into the relay ([install chisel](https://github.com/jpillora/chisel#install) locally, e.g. `brew install chisel-tunnel`):

```bash
varlock proxy start --port 8080 --cert-dir ./proxy-certs
# in another terminal
chisel client --auth "$TUNNEL_AUTH" "https://$RELAY_HOST" R:8080:127.0.0.1:8080
```

Agent sandboxes are wired exactly as in the [broker recipe](#start-agent-sandboxes), pointing their chisel client at `relayHost` and using the CA certs from your local `./proxy-certs`. Requests from agents now resolve against your local schema, the interactive daemon's live request log and [hot reload](/guides/proxy/#editing-the-schema-while-a-session-is-running) work as usual, and `varlock proxy audit` on your machine shows the fleet's traffic.

### Alternative: E2B request transforms

E2B has a beta feature ([`network.rules`](https://e2b.dev/docs/sandbox/internet-access)) that injects headers into egress requests for matching hosts at their gateway, so a sandbox can call an API without holding the key. The difference is custody and control: with `network.rules` you hand the real secret to E2B's control plane, and the injection rule is static per sandbox. With the varlock proxy, secrets stay in the broker (or on your machine), and you keep schema-driven policy, path and method rules, strict egress, [hot reload](/guides/proxy/#editing-the-schema-while-a-session-is-running), response scrubbing, and a per-request audit log. If all you need is one static header on one host and you are comfortable with E2B holding the key, `network.rules` is less machinery; past that, use the proxy.

## Limitations

- The proxy data plane has no client authentication of its own yet; reachability is the gate. In these recipes that gate is the chisel tunnel's auth, and the proxy itself never listens off loopback. Do not expose the proxy port through a raw public TCP tunnel.
- Interactive daemon features (the live request log, pressing `r` to reload) need a human at the proxy's TTY: available in the [proxy-on-your-machine](#proxy-on-your-machine) shape, not in the broker shape.
- A broker sandbox is a single point of failure for its fleet. Manage its lifetime explicitly (`timeoutMs`, or E2B's pause/resume) and let agent chisel clients reconnect if it restarts.
- Domain-based `allowOut` rules match by SNI and Host header, per E2B's docs; IP-literal traffic needs CIDR rules.
