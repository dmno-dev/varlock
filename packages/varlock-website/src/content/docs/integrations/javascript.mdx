## Node.js - `varlock/auto-load`

The best way to integrate varlock into a plain Node.js application (⚠️ version 22 or higher) is to import the `varlock/auto-load` module. This uses `execSync` to call out to the varlock CLI, sets resolved env vars into `process.env`, and initializes varlock's runtime code, including:

- varlock's `ENV` object
- log redaction (if enabled)
- leak detection (if enabled)

```js title="example-index.js"
import 'varlock/auto-load';
import { ENV } from 'varlock/env';

const FROM_VARLOCK_ENV = ENV.MY_CONFIG_ITEM; // ✨ recommended
const FROM_PROCESS_ENV = process.env.MY_CONFIG_ITEM; // 🆗 still works
```

:::note[dotenv drop-in replacement]
If you are using [`dotenv`](https://www.npmjs.com/package/dotenv), or a package you are using is using it under the hood - you can seamlessly swap in varlock using your package manager's override feature. See the [migrate from dotenv](/guides/migrate-from-dotenv) guide for more information.
:::


## Boot via `varlock run`

A less invasive way to use varlock with your application is to run your application via [`varlock run`](/reference/cli-commands/#run).

```bash
varlock run -- <your-command>
```

This will load and validate your environment variables, then run the command you provided with those environment variables injected into the process. This will not inject any runtime code, and varlock's `ENV` object will not be available.

If you have installed varlock as a project dependency instead of globally, you should run this via your package manager:
<ExecCommandWidget command="varlock run -- <your-command>" showBinary={false} />

In `package.json` scripts, calling `varlock` directly will work, as your package manager handles path issues:
```json title="package.json"
"scripts": {
  "start": "varlock run -- node index.js"
}
```

Even when using a deeper integration for your code, you may still need to use `varlock run` when calling external scripts/tools, like database migrations, to pass along resolved env vars.

:::tip[Setting the current environment]
Varlock can load multiple environment-specific `.env` files (e.g., `.env.development`, `.env.production`) by using the [`@currentEnv` root decorator](/reference/root-decorators/#currentenv) to specify which env var will set the current environment.

If not using the default (usually `development`), you'll want to pass it in as an environment variable when running your command.

```bash
APP_ENV=production varlock run -- node index.js
```

See the [environments guide](/guides/environments) for more information about how to set the current environment.
:::



## Front-end frameworks

While environment variables are not available in the browser, many frameworks expose some env vars that are available _at build time_ to the client by embedding them into your bundled code. This is best accomplished using tool-specific integrations, especially for frameworks that are handling both client and server-side code.


:::tip[Isomorphic env vars]
The `varlock/env` module is designed to be imported on both the client and server, so frameworks that run code in both places (like Next.js) can import it.
:::

## When things go wrong

### `Unable to find varlock executable` in a monorepo sub-package

`varlock/auto-load` calls the CLI via `execSync` during module initialization. It searches for `node_modules/.bin/varlock` starting from the installed `varlock` package directory (not just `process.cwd()`), so sub-packages like `apps/web` can find a dependency installed locally even when your shell cwd is the workspace root.

If you still see `Unable to find varlock executable`:

- Ensure `varlock` is a dependency of the package that imports `varlock/auto-load`, not only at the monorepo root
- Run the app via your package manager (`pnpm --filter web dev`) so `node_modules/.bin` is on `PATH`, or boot via [`varlock run`](/reference/cli-commands/#run) instead
- Install the [standalone binary](/getting-started/installation/#as-a-standalone-binary) globally as a fallback

### `varlock/auto-load` fails on Vercel / serverless

Serverless bundlers often trace `node_modules/varlock/dist` without including `node_modules/.bin/varlock`. During cold start, `auto-load` then fails before your handler runs with `Unable to find varlock executable`.

Recent varlock releases include a package-local CLI fallback so `auto-load` can run the CLI entry bundled inside the `varlock` package itself, without depending on `.bin` being present. Upgrade to the latest varlock if you hit this on an older version.

**Workarounds if you still see this error:**

- Prefer [`varlock run`](/reference/cli-commands/#run) or a framework integration that loads env at build/deploy time instead of `auto-load` at runtime
- Ensure your bundler includes `node_modules/.bin/varlock` if you must use `auto-load` in serverless

### When to use `varlock run` vs `varlock/auto-load`

| | `import 'varlock/auto-load'` | `varlock run -- <command>` |
|---|---|---|
| **Runtime features** | `ENV` object, log redaction, leak detection | Env vars only — no runtime patches |
| **Node.js version** | Requires Node 22+ | Any process varlock can spawn |
| **Best for** | Long-running Node apps where you want the full runtime | CLI tools, migrations, non-JS/TS apps, serverless, or one-off scripts |
| **Monorepo / bundler edge cases** | Depends on finding or bundling the CLI at import time | Uses the `varlock` binary you invoke directly |

Use **`auto-load`** when you want varlock's runtime in your Node process and you control how the app is started (local dev, containers, traditional servers).

Use **`varlock run`** when you only need resolved env vars injected into a child process, when `auto-load` cannot find or bundle the CLI, or when you are wrapping external tools that read `process.env`.
