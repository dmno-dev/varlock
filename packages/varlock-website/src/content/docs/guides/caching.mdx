Varlock supports caching for resolver results (via [`cache()`](/reference/functions/#cache)) and for plugin authors (via plugin cache APIs). Caching can improve performance and reduce external API calls, but the right mode depends on your environment and threat model.

## Quick mental model

There are two broad caching patterns in the ecosystem:

- **Varlock cache store**: used by `cache()` and plugin persistent cache APIs. This is controlled by global [`@cache`](/reference/root-decorators/#cache) mode.
- **Plugin in-session memoization**: some plugins keep an internal in-memory map for a single resolution pass. This is plugin-specific and separate from `@cache`.

## Global cache modes (`@cache`)

Set cache behavior globally in your file header:

```env-spec title=".env.schema"
# @cache=auto
# ---
```

Supported values:

- `auto` (default): varlock chooses mode automatically
- `memory`: cache in-process only (not persisted)
- `disk`: cache in encrypted disk storage (persists across invocations)
- `disabled`: disable cache reads/writes globally
- `true`: alias for `disk`
- `false`: alias for `disabled`

Auto mode policy:

- Uses `memory` in CI
- Uses `memory` when local encryption is using file-based fallback
- Uses `disk` otherwise

## Choosing a mode

Recommended defaults:

- **Local dev on trusted machine**: `@cache=auto` (usually ends up as `disk`)
- **CI / ephemeral automation**: `@cache=memory` (or keep `auto`)
- **Deployed env with file-based local-encryption fallback**: `@cache=memory`
- **High-sensitivity workflows**: `@cache=disabled`

## Per-invocation CLI controls

Both [`varlock load`](/reference/cli-commands/#load) and [`varlock run`](/reference/cli-commands/#run) support:

- `--skip-cache`: disables cache reads and writes for that invocation (overrides `@cache`)
- `--clear-cache`: clears the active cache store before resolving values

If both are used, cache is cleared first, then skipped for the rest of the run.

## Inspecting and clearing cache

Use [`varlock cache`](/reference/cli-commands/#cache) to inspect and clear the **disk cache**:

```bash
varlock cache status
varlock cache clear --yes
varlock cache clear --plugin 1password --yes
```

Notes:

- This command manages disk cache entries only.
- In `memory` mode, entries are process-local and are not visible via `varlock cache`.

## Plugin cache behavior

Plugins that expose cache TTL controls (for example, 1Password `cacheTtl`) still respect global cache mode:

- `@cache=disk`: plugin cached values persist across runs
- `@cache=memory`: plugin cached values are in-process only
- `@cache=disabled` or `--skip-cache`: plugin persistent cache is bypassed

Plugin-specific in-session memoization may still occur even when persistent caching is disabled.

## Cache key guidance

When using `cache(..., key="...")` or plugin cache keys:

- Keep keys stable and deterministic
- Avoid control characters
- Keep keys reasonably short
- Namespace keys (for example `org:service:token`) to avoid collisions

Varlock validates cache keys and throws on invalid key text (for example empty strings, control chars, overly long keys).

## Troubleshooting

### "I changed a resolver but still see old values"

- If you used a custom `key`, that key may intentionally pin the cache entry.
- Run with `--clear-cache` or clear entries via `varlock cache clear`.

### "`varlock cache` shows nothing, but caching seems active"

- You are likely in `memory` mode (for example in CI or file-backend fallback mode).

### "I want no cache artifacts on disk"

- Use `@cache=memory` (or `@cache=disabled`) and avoid `disk` mode.

### "I need deterministic fresh fetches in CI"

- Run with `--skip-cache` for that job/stage.
